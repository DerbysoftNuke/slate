概述

Go 渠道 API 是德比软件对酒店销售渠道定制开放的一整套后端接口，包括酒店预定、取消、查询、验价，数据实时缓存和酒店内容服务。酒店销售渠道可通过此套接口实现与任何已有酒店供应商的对接，并且大大减少增加新的供应商对接所需要的成本。本 API 开发文档适用于JSON格式，详细说明了如何利用这些接口来完成酒店和渠道的对接，以及各种注意事项，可以作为实施工程师开发 Go API 的技术指南。

初次使用Go

当渠道已经与德比软件签订了通过Go API进行对接的合同之后，渠道根据我们提供的API接口文档和相关测试环境的信息可以开始进行开发和验证。

下面的步骤只用实施一次无论一个渠道希望对接多少个酒店供应商。

  • 德比软件实施人员会将Go的实施调查问卷提供给渠道。
  • 渠道完成调查问卷并提交给德比。
  • 德比软件的实施人员会通过邮件将测试环境相关信息发送给渠道。
  • 渠道开发完成并完成自测。
  • 德比软件实施人员确认测试用例并通知渠道相关结果。
  • 渠道将他们已经签订合同的供应商通知德比软件，并确认可以开始三方测试。
在此过程中遇到任何问题，德比测试工程师将协调解决

当渠道的认证过程通过之后，三方测试就可以开始了。一个渠道无论对接多少供应商，都需要首先进行三方测试，这是为了保证订单流程畅通。三方测试一般通过电话会议的方式进行。当所有测试流程走完、结果达到预期之后，就可以着手准备上线酒店和展示内容了。 生产环境开始运作之后，如果遇到问题，可以联系 Go.support@derbysoft.net。

权限与认证

  • 德比软件会为每个接入的渠道分配一个唯一的访问令牌。任何一个请求，只有在 HTTP 头里包含该令牌，才能访问德比软件提供的API服务。
  • 出于性能考虑，可以使用 GZIP 对 SOAP 消息进行解压缩
  • 目前可以识别的字符集是 UTF-8，内容类型是 text 或 JSON (JavaScript Object Notation)

HTTP 请求头实例:

HTTP Request Header POST /json/availability HTTP/1.1 Authorization: 53ac07777cdffac2d53000002d698728ce964432d7167596bc005c5fc Accept-Encoding: GZIP Content-Type: text/json;charset=utf-8

HTTP 响应头实例:

HTTP Response Header POST /json/availability HTTP/1.1 Authorization: 53ac07777cdffac2d53000002d698728ce964432d7167596bc005c5fc Accept-Encoding: GZIP Content-Type: text/json;charset=utf-8

Web 服务的一些限制

  • 目前 Web 服务仅支持英文语言
  • 所有消息均采用 UTF-8 字符集表示
  • 所有描述字段包括注释、名字、姓氏、取消理由、税费名称等均采用拉丁字符
  • 如果来自渠道的订单上出现非拉丁字符，德比可以提供字符格式转换。如您需要转换服务，请告知我们
  • 德比限制服务请求量。例如，批量查看房源服务限制每日请求数为200万个，最大吞吐量（TPS）为 100。一旦超出这个限制，用户将会收到 Http 429 太多请求的报错
  • 出于 IP 范围限制和负载平衡考虑，每个渠道用户最多可被分配 5 个 IP 用于下单。一般情况下，出于安全原因，渠道仅使用1至 2 个 IP 将订单信息发送至 Go。

入口地址和域

德比为不同请求类型提供了不同的入口地址。一般情况下，德比为所有请求提供了两个域，*.derbysoftsec.com 和 *.derbysoftca.com。酒店缓存数据接口、内容接口和查看房源数据通过*.derbysoftsec.com接入。 其他和订房相关的接口通过 *.derbysoftca.com接入。

酒店数据缓存接口

酒店缓存接口提供了高性能，稳定可靠的数据缓存服务，渠道可以通过调用此接口来实现自己的数据缓存或者直接用于数据展示。

最佳实践为首先使用ChangeDiscovery来检查在某个日期区间内，某些酒店是否有数据变化，然后通过将变化的酒店和日期作为参数再调用MultipleAvailability接口获取具体的数据

Multiple Availability

  • URL - https://go.derbysoftsec.com/go/json/multipleavailability/yourusername/
  • Method - POST

The shopping transaction (or multiple availability) is expected to be used as the first request for available properties. This is the REQUIRED transaction for shopping rates. Even though it is called a multi-property availability, a single property can be sent in this message. The intent and design of this message is to facilitate a property search page display or a property search availability cache with very short response times. This message should be used for all availability requests other than the live availability immediately just prior to a reservation. The shopping message is used to get rates from a cache (if there is a cache). The request includes the check-in date, stay nights, guest occupancy, the request rooms and hotel codes. The response to the multi property availability request returns the availability only for the room types and rates available for the requested dates. The rates can vary depending on the dates and room types. If there is no availability for a property, room type, and/or rate plan, that item will not be returned in the message. 此接口根据请求条件返回可用的酒店数据，包括可用的房型、价格计划、售价、取消政策、税费、早餐等。请求的条件包括：入住日期、离店日期、需要的房间数量、成人小孩数量及小孩年龄。如果没有符合条件的数据，将会返回空的应答，类似于：

Change Discovery

  • URL - https://go.derbysoftsec.com/go/json/changediscovery/yourusername/
  • Method - POST

The Change Discover API allows you to be more efficient in creating and maintaining a local cache. Please notice, the max date range in EACH request is 31, if your request is beyond the max data range, you can send multiple request. For example, request 1-30 in one request, request 31-60 in another request, etc.

Best Practice

  • Use different interval for different date range. For example,
    • Every 30 minutes for 1-30 days.
    • Every 60 minutes for 30 - 90 days (You can send 2 requests with each request maximum date range 31.).
    • Every 2 hours for 90-180 days (Similar with above).
    • Every 8 hours for 180-360 days or just pass request to DerbySoft.

Reservation API

Reservation API allows you to do live check, make reservation, cancel reservation and query reservation. The normal process is: 1. You should do live check before you send reservation request, to gain the latest rate and availability information and compare with the cache data. 2. Then you can send Reservation request to DerbySoft, notice that we have two booking models: 1. Booking Request Model (ForceSell = false) * Booking confirmed by hotel. Distributor accepts hotel rates. 2. Booking Notification Model ( ForceSell = true) - Not Support Now * Booking confirmed by distributor. Hotel accepts distributor rates. 3. You can cancel your reservation by using CancelReservation API. 4. Or query the status of one specific reservation through QueryReservation API.

Appendix

Cancellation Policy

Cancellation policy defines what penalty will be charge when a guest cancels the booking at certain advance time range. The penalty is related to no show or a time range before no show.

Time Range

  • No Show. The customer doesn't cancel the booking or cancel it after arrival day. Most of hotel define the no show time is 12 AM -10 AM of next day of arrival. The last penalty without advance day is the penalty of no show.
  • Deadline on arrival day- No Show. Many hotels have a deadline on arrival day and will apply different penalty when customer cancel the booking after the deadline. The format of deadline is 4 PM or 6:30 PM. The default deadline is 12 AM.
  • Advance day. The format is 1D, 2D, 7D or AD. 1D means 24 hours before 'Deadline'. AD means unlimited advance day.
  • Timezone. The timezone of the deadline or advance day is hotel local timezone.

Penalty Type

There are three penalty types.

  • Percentage. Will charge the percentage of the whole stay. For example, 80P, 100P
  • Amount. Will charge fix amount. The currency of this amount is same as rate. For example, 120, 245.75
  • Nights. Will charge the fee of nights. For example, 1N, 2N.

Examples

The CheckIn day of below examples is 2016-10-15.

A Full Example

AD20_7D50_3D1N_1D50P_6PM0D80P_100P

  • 100P is the penalty of no show. It means 100 percent of the whole stay will be charged if customer doesn`t cancel the booking and not arrive at hotel.
  • 6PM0D80P defines the penalty between 6 PM of arrival day(0D) and no show. It means 80 percent of whole stay will be charge if you cancel the booking between 6 PM and 11:59 PM of 2016-10-15.
  • 1D50P defines the penalty between 2016-10-14 6 PM and 2016-10-15 6 PM. It means 50 percent of the whole stay will be charged if you cancel the booking between 2016-10-14 6 PM and 2016-10-15 6 PM.
  • 3D1N defines the penalty between 2016-10-12 6 PM and 2016-10-14 6 PM. It means first night will be charged if you cancel the booking between 2016-10-12 6 PM and 2016-10-14 6 PM.
  • 7D50 defines the penalty between 2016-10-08 6 PM and 2015-10-12 6 PM. It means 50$ will be charged if you cancel the booking between 2016-10-18 6 PM and 2016-10-12 6 PM.
  • AD20 defines the penalty between booking time and 2015-10-12 6 PM. It means 20$ will be charged if you cancel the booking between the time you make the reservation and 2016-10-12 6 PM.

  • Exclusive - It means the fee isn't included in AmountBeforeTax.

  • PER_PERSON_PER_NIGHT - The Amount will be charge every adult in the room and every night.

  • PER_ROOM_PER_STAY - The Amount will be charge every room for the whole stay.
  • PER_PERSON_PER_STAY - The Amount will be charge every adult in the room for the whole stay.

Fee Change

One tax could change by government. For example,

  • CheckIn: 28 Dec
  • CheckOut: 3 Jan
  • Tax1: City Tax 2$ PER_ROOM_PER_NIGHT
  • Tax2: State Tax 8.5%
  • Daily Rate: 100$

The state tax changes from 8.5% to 9.5% next year. There are 2 possible calculation logic of above stay:

  • AmountAfterTax of day before 1 Jan : 100$ * (1+8.5%) + 2$ = 110.5$. AmountAftaxTax of day after 31 Dec = 100$ * (1+9.5%) + 2$ = 111.5$. Total Aftre Tax = 110.5$ * 4 + 111.5 * 2 = 665.0$

We return EffectiveDate and ExpireDate on fee so you know there is a tax change. There are no EffectiveDate and ExpireDate on fees if there is no tax change.

Extra Person Charge

Currently, only IHG returns ExtraPersonCharge and ExtraPersonChargeAfterTax. We pass this information to you because IHG extra person charge is non-markupable and non-commisionable. You can not do correct markup or commission calculation without this information.

For example, bar rate details are:

  • AmountBeforeTax : 120$
  • AmountAfterTax : 133$
  • ExtraPersonCharge : 20$
  • Fees : 10% state tax and 1 $ per room per night city tax.
ExtraPersonCharge is the total ExtraPersonCharge. For example, the ExtraPersonCharge is 40$ for 4 adults(2 extra person).

Markup

The discount that IHG gives to you is 20%. So the rate details you will get:

  • AmountBeforeTax: 100$ = (AmountBeforeTax of Bar - ExtraPersonCharge)*(1-discount) + ExtraPersonCharge = (120$ - 20$) * (1-20%) + 20$
  • AmountAfterTax: 111$ = AmountBeforeTax * (1+state tax) + city tax = 100$*(1+10%) + 1$
  • ExtraPersonCharge: 20$
  • ExtraPersonChargeAfterTax: 22$ = ExtraPersonCharge * (1+ state tax) = 20 *(1+10%)
  • Fees: 10% state tax and 1 $ per room per night city tax.

If you want to markup the rate to Bar rate, you need to do below calculation:

  • AmountBeforeTax of Bar : (AmountBeforeTax - ExtraPersonCharge) / (1 - discount) + ExtraPersonCharge = (100$ - 20$) / (1 - 20%) + 20$ = 120$
  • AmountAfterTax of Bar : Apply fees on AmountBeforeTax of bar = 120$ *(1 + 10%) + 1 = 133$

If you don't need to apply the percentage tax to markup, you can do a easy way below:

  • AmountBeforeTax after markup : (AmountBeforeTax - ExtraPersonCharge) / (1 - discount) + ExtraPersonCharge = (100$ - 20$) / (1 - 20%) + 20$ = 120$
  • AmountAfterTax after markup : AmountBeforeTax after markup + (AmountAfterTax - AmountBeforeTax) = 120$ + (111$ - 100$) = 131$

Commission

The commission that IHG gives to you is 10%. The commission calculation is:

  • Commission Amount : (AmountBeforeTax - ExtraPersonCharge) * Commission = (120$ - 20$) * 10% = 10$
