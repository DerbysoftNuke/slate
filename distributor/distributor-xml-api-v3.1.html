Overview

Introduction

Go is a product produced by Derbysoft that includes a set of web services for online hotel data extraction and distribution. For each Go web service, an API is designed for travel distribution partners. Using these APIs, the partners are able to access availability, rates, property content, product details for hotel companies worldwide, as well as to send reservation and cancellation transactions. With Go APIs, partners can access travel data from any existing travel service supplier. The cost of developing Interface for new distributors using Go is greatly reduced.

This document is a development and implementation guide to DerbySoft’s Go Web Services. The transaction flow is documented in the following diagrams.

Process

The section introduces the processes for deployment and going live. Experienced users may skip it.

After the Distributor signs the contract to connect with the Go API to DerbySoft, the self-certification process can be started. This is to ensure the connection between the Distributor and DerbySoft is working smoothly. The following steps are only carried out once no matter how many Suppliers the Distributor would like to connect to.

  • DerbySoft Implementations provides the Go Distributor Implementation Questionnaire to the Distributor
  • Distributor fills out the questionnaire and submits it
  • DerbySoft Implementations shares the test environment credentials (Username+ Password) by email
  • Distributor works on the API and completes the Self-Certification Form
  • DerbySoft Implementations verifies the test scenarios and informs the Distributor whether the scenarios are valid
  • Distributor provides the list of suppliers they have contracts with and confirms that the 3-way testing can be started
For any questions regarding the process or the API development, distributors are recommended to find answers from Go Development Platform. For the questions whose answer cannot be found from the platform, Pre-launch Support Team can be contacted via go.dev@derbysoft.com.

Once the self-certification is completed, the 3-way testing phase can be started. This will be repeated for each new Supplier to ensure the correct availability and booking flows work properly. The testing is normally carried out via teleconference with all 3 parties. When all test scenarios are completed, and the results are as expected, the pilot hotels and the remaining rollout can be planned for going live.

After the production environment comes into use, distributors may contact the Support Team for further queries via Go.support@derbysoft.net.

Authentication

The Access Token is an encryption key provided by DerbySoft to distributors for accessing APIs. Authentication is required when distributors access Derbysoft's endpoints. Here is an example of HTTP request header:

POST /xml/availability HTTP/1.1
Authorization: 53ac07777cdffac2d53000002d698728ce964432d7167596bc005c5fc
Accept-Encoding: gzip
Content-Type: text/xml;charset=utf-8

Note that GZIP needs to be utilized for decompressing SOAP messages.

Web Service Method

Go APIs provide web service methods including request and response messages. Distributors will be granted credentials (username+password) to access these APIs via SOAP (Simple Object Access Protocol) version 1.1.

If you would like to transact using a different protocol or a different version of SOAP, please contact Go.dev@derbysoft.net.

WSDL (Web Services Description Language)

WSDL is used for code generation by distributors.

  • For Content APIs: http://uatgo.derbysoftsec.com/v3_1/hotel/content/propertySearch.wsdl

Endpoint and Domain

DerbySoft provides different endpoInts for each request type. Generally, DerbySoft provides two domains for all requests, which are:

  • Domain *.derbysoftsec.com for shopping, live availability, content APIs

  • Domain *.derbysoftca.com for booking APIs.

Limitations

The limitations on web services are as below:

  • English is the only language supported now.
  • UTF-8 is the only charset supported by messages.
  • Latin characters are supported by description fields such as first name, last name, cancellation description, fee name, etc.
  • Character romanization service can be provided by DerbySoft if distributor sends non-latin characters on reservation request. Please let us know if you need this service.
  • Volume for each request is limited. For example, volume limit for Multiple Availability is 2 million requests per day, and the maximum TPS is 100. 'Http 429 - Too Many Requests' error will be returned when the limitations are exceeded.
  • 5 IPs can be allocated to one distributor as a maximum for reservations due to limitation of IP or IP range of ELB (Load Balance). Normally, for security reasons, only 1 to 2 IPs are used by each distributor for sending reservation data to GO.
  • HTTPS for all requests and only TLS 1.2 is supported.

Token

DerbySoft supports token codes/echo tokens to identify unique message sets. These tokens are generated by the Distributor, and DerbySoft returns the request/response with the same token for easy identification. The tokens can be alphanumeric with less than 64 characters.

Token codes for multi-availability requests, single availability requests (live availability check), booking requests, and cancellation requests need to be unique per transaction.

Shopping API

Two shopping APIs are provided: Change Discovery and Multiple Availability Check.

Distributors are recommended to:

  • first, filter the hotels whose shopping rates are changed with Change Discovery, and
  • then, request the availability data for the changed hotels only by providing the hotel code and the date indicating an ARI change with Multiple Availability Check

This process will help avoid resource constraints caused by requesting all hotel availability data every time.

Change Discovery

  • URL - https://go.derbysoftsec.com/go/v3.1/xml/changediscovery/yourusername/
  • Method - POST

Change Discovery provides reliable data caching service with high performance. With this resource, distributors can develop their own data cache or illustrate data directly.

The Change Discovery API allows you to be more efficient in creating and maintaining a local cache. Start Date and End Date will be passed through the request and the response will return the hotels whose ARI has changed between the Start Date and End Date specified.

Note that the maximum day range supported by a single request is 31 days. To deal with day ranges greater than 31 days, multiple requests can be used. For example: Use Start and End dates representing days 1-30 in one request, and dates representing days 31-60 in another request, etc.
Please note that a search by city is not possible due to cities and their parameters being defined differently by each Distributor and Supplier.

One or more requests can be used for exacting data with different date ranges. For example

  • For 1 - 30 days, send requests every 30 minutes
  • For 30 - 90 days, send 1 - 3 requests each time
  • For 90 - 180 days, send 3 - 6 requests each time
  • For 180 - 360 days, send 6 - 12 requests each time, or pass your business request to us

Distributors can filter the list of changed hotels with Change Discovery and then extract data on changed hotels using the Multi-Property Availability. Refer to the Appendix for an example of a Rate Changed Hotel only request.

Multiple Availability

  • URL - https://go.derbysoftsec.com/go/v3.1/xml/multipleavailability/yourusername/
  • Method - POST
Please note that the name of the request is Multi-Property Availability Request, however this request can be used for individual and multiple hotels.

The multi-property availability request is a shopping transaction request and should be used to request rate and availability for an individual hotel or a selection of up to 20 hotels. This request hits the DerbySoft cache and should be used for all rate and availability inquiries as well as to fill the Distributor’s internal cache, if applicable.

In the multi-property avail request, it is possible to request rates and availability for several hotels, guest counts and room types. Check-in date, stay nights, guest occupancy, requested rooms, and hotel codes can be set as criteria in the request.

Please note that the hotel codes are ALWAYS prefixed. Normally, it is prefixed with brandcode such as Brandcode-Hotelcode. For wholesales, the hotel codes are prefixed with wholesaler code such as Wholesalercode-Hotelcodes.

The response to the multi-property availability request returns the availability only for the room types and rates available for the requested dates. The rates can vary depending on the dates and room types. If there is no availability for a property, room type, and/or rate plan, that item will not be returned in the message.

With caching the availability and rate data in advance, searching response time for related info is greatly reduced. It is strongly recommended to use a Multi-Property Availability Check API before completing a reservation.

Reservation API

Go Reservation includes 4 APIs which are Live Availability Check, Reservation, Cancellation, and Query Reservation. In production, Distributors may need privileges to access the above APIs except for Live Availability Check.

Please provide your IP to Go Implementation Analysts. You will be allowed to access the Reservation APIs after we add your IP Into the whitelist. By default, each distributor can be allocated 5 IPs in production. Please contact us if you need more IPs.

Live Availability Check

  • URL - https://go.derbysoftsec.com/go/v3.1/xml/singleavailability/yourusername/
  • Method - POST

The Live Availability Check message is used do a last minute availability check to the Supplier CRS to ensure the rate and room type in the Booking request is available for booking. It is required to be invoked before customers making reservations. The Live Availability Check is only supported for one individual hotel per request.

The request is sent to

  • the supplier when the Supplier is configured with the Live Availability Check method
  • Derbysoft cache for the Suppliers not support Live Availability Check Method
Please note that if a Live Availability Check request is for multiple rooms at a hotel, the multiple rooms needs to be of the same room type and the same guest count.

The response to the Live Availability Check includes the stay dates, room type, guest count, price and the available rate plans. Adult count, kids count, and sometimes kids' age can make the search result different. Then, the available room numbers, room types, rate plans, cancellation policy, etc. will be returned.

Reservation

  • URL - https://go.derbysoftca.com/go/v3.1/xml/reservation/yourusername/
  • Method - POST

The Distributor sends the Booking Request only after sending the Single Availability Request and receiving availability back from the Supplier. The Booking Request is passed via DerbySoft to the Supplier CRS and is immediately confirmed or declined by the Supplier.

It is possible to book several rooms in one Booking Request, however, there are some rules around multi room bookings:

  • Multiple rooms can be booked in one request only when they are with the same room type, rate plan, and guest count
  • Multiple rooms bookable in one request can only be booked for the number of rooms the supplier defines. This number of rooms might vary.

You can book multiple rooms in a single MakeReservation RQ by using the Number of Units attribute. However, all rooms must be the same room type, rate plan type, and guest count configuration to be included in one request. If any of those need to be different, for example you need one room with a King bed and one room with 2 Doubles, then you would need to do two separate requests, one for each room.

Please note that if a Distributor wants to make reservations for example for a Double and a Triple room, 2 Booking Requests need to be sent.

After the request is transacted by the supplier, the booking confirmation and the confirmed price are returned. Some suppliers may return extra information such as cancellation policy, tax rules, etc.

There are different reasons the Booking Request can fail. In case of failure, it is recommended to check whether the data transmission from Derbysoft to the Distributor is normal via Query API.

  • If the data transmission is fine, the most common failure reasons are that the Booking Request contains false data. Please ensure that the credit card details, room type code and rate code are correct.
  • Due to synchronous mode being adopted in reservation request handling, the request can be blocked when the response time exceeds the time out threshold of the API. Refer to Time Out section for the Time Out thresholds for different APIs defined by Derbysoft. If distributors need to define their own time out, it is recommended that the value smaller than the that defined by Derbysoft, or the thresholder defined by Derbysoft will be adopted.
    If there is issue in data transmission, please contact Derbysoft Support Team for update booking data manually.

Time Range

  • No Show. The customer doesn't cancel the booking or cancel it after arrival day. Most of hotel define the no show time is 12 AM -10 AM of next day of arrival. The last penalty with NoShow="true" is the penalty of no show. The penalty MUST and ONLY has one.
  • Deadline on arrival day - No Show. Many hotels have a deadline on arrival day and will apply different penalty when customer cancel the booking after the deadline. The format of deadline is 4PM or 6:30PM. Deadline is mandatory if there are not only no-show penalty.
  • Advance Time Window. Defined by MaxHoursBeforeCheckIn and MinHoursBeforeCheckIn. They are hours before 'Deadline'. MaxHoursBeforeCheckIn="9999" means unlimited advance day. Currently, hours MUST be multiple of 24 hours. They are days.
  • The sequence of time range is bigger advance day -> small advance day -> deadline on arrival -> no show.

Timezone

The timezone of the deadline or advance day is hotel local timezone. There are exceptions for few supplier(most of them are wholesales). You may need to convert them to your own timezone.

Penalty Type

There are three penalty types. One CancelPenalty one penalty type.

  • Percentage. Will charge the percentage of the whole stay. For example, <CancelPenalty NoShow="true" Percent="100"/>
  • Price. Will charge fix amount. The currency of this amount is same as rate. For example, <CancelPenalty MaxHoursBeforeCheckIn="48" MinHoursBeforeCheckIn="0" NoShow="false" Price="120"/>
  • Nights. Will charge the fee of nights. For example, <CancelPenalty MaxHoursBeforeCheckIn="48" MinHoursBeforeCheckIn="0" NoShow="false" Nights="1"/>.

The ChargeType of one fee is unique. We return same ChargeType for same fee. The type of Fee should be fixed amount or percentage. They both should not be present in one Fee. So there is Amount with Unit or Percent for one Fee node.

Inclusive & Exclusive

Type could be:

  • Inclusive - It means the fee is included in AmountBeforeTax.
  • Exclusive - It means the fee isn't included in AmountBeforeTax.
  • Cumulative - A special percentage tax and Exclusive. We don't return Cumulative and Inclusive fee.

Amount Fee

The currency of amount is same as rate. There are 4 units of fixed amount tax. Unit is required if Amount isn't null.

Per Night

  • PER_ROOM_PER_NIGHT - The Amount will be charge every room and every night.
  • PER_PERSON_PER_NIGHT - The Amount will be charge every adult in the room and every night.

Per Stay

  • PER_ROOM_PER_STAY - The Amount will be charged every room for the whole stay.
  • PER_PERSON_PER_STAY - The Amount will be charged every adult in the room for the whole stay.

Percent Fee

Percent is quite simple. It's the percentage of the rate on each day.

Cumulative Fee

A general case of cumulative fee is service charge is taxable in some state. For example,

AmountBeforeTax = $100

Cumulative Fees

<Fees>
  <Fee ChargeType="Service Charge" Type="Exclusive" Percent="5">
  <Fee ChargeType="State Tax" Type="Cumulative" Percent="10">
  <Fee ChargeType="City Tax" Type="Exclusive" Amount="1" Unit="PER_ROOM_PER_NIGHT">
</Fees>

Calculation is:

  • AmountAfterServiceCharge = AmountBeforeTax *( 1 + Service Charge) = $100 * (1+5%) = $105
  • AmountAfterStateTax = AmountAfterServiceCharge * ( 1+ State Tax) = $105 *(1 + 10%) = $115.5
  • AmountAfterTax = AmountAfterStateTax + City Tax = $115.5 + $1 = 116.5

Fee Change

One tax could change by government. For example,

  • CheckIn: 28th Dec.
  • CheckOut: 3rd Jan.
  • Tax1: City Tax $2 PER_ROOM_PER_NIGHT
  • Tax2: State Tax 8.5%
  • Daily Rate: $100

The state tax changes from 8.5% to 9.5% next year. There are 2 possible calculation logic of above stay:

  • AmountAfterTax of day before 1 Jan : $100 * (1+8.5%) + $2 = $110.5. AmountAftaxTax of day after 31 Dec = $100 * (1+9.5%) + $2 = $111.5. Total Aftre Tax = $110.5 * 4 + 111.5 * 2 = $665.0

We return EffectiveDate and ExpireDate on fee so you know there is a tax change. There are no EffectiveDate and ExpireDate on fees if there is no tax change.

Extra Person Charge

Currently, only IHG returns ExtraPersonCharge and ExtraPersonChargeAfterTax. We pass this information to you because IHG extra person charge is non-markupable and non-commisionable. You can not do correct markup or commission calculation without this information.

For example, bar rate details are:

  • AmountBeforeTax : $120
  • AmountAfterTax : $133
  • ExtraPersonCharge : $20
  • Fees : 10% state tax and $1 per room per night city tax.
The ExtraPersonCharge stands for ExtraPersonCharge BeforeTax. ExtraPersonCharge is the total ExtraPersonCharge. For example, the ExtraPersonCharge is $40 for 4 adults(2 extra person).

Markup

The discount that IHG gives to you is 20%. So the rate details you will get:

  • AmountBeforeTax: $100 = (AmountBeforeTax of Bar - ExtraPersonCharge)*(1-discount) + ExtraPersonCharge = ($120 - $20) * (1-20%) + $20
  • AmountAfterTax: 111$ = AmountBeforeTax * (1+state tax) + city tax = $100*(1+10%) + $1
  • ExtraPersonCharge: $20
  • ExtraPersonChargeAfterTax: $22 = ExtraPersonCharge * (1+ state tax) = 20 *(1+10%)
  • Fees: 10% state tax and $1 per room per night city tax.

If you want to markup the rate to Bar rate, you need to do below calculation:

  • AmountBeforeTax of Bar : (AmountBeforeTax - ExtraPersonCharge) / (1 - discount) + ExtraPersonCharge = ($100 - $20) / (1 - 20%) + $20 = $120
  • AmountAfterTax of Bar : Apply fees on AmountBeforeTax of bar = $120 *(1 + 10%) + 1 = $133

If you don't need to apply the percentage tax to markup, you can do a easy way below:

  • AmountBeforeTax after markup : (AmountBeforeTax - ExtraPersonCharge) / (1 - discount) + ExtraPersonCharge = ($100 - $20) / (1 - 20%) + $20 = $120
  • AmountAfterTax after markup : AmountBeforeTax after markup + (AmountAfterTax - AmountBeforeTax) = $120 + ($111 - $100) = $131

Commission

The commission that IHG gives to you is 10%. The commission calculation is:

  • Commission Amount : (AmountBeforeTax - ExtraPersonCharge) * Commission = ($120 - $20) * 10% = $10
